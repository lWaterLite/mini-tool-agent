# 子模块 4：手写最小 Agent Loop 概念教学

对应学习路线：模块 1「LLM 工程基础与最小 Agent」  
学习主题：Agent Loop、模型决策、工具调用、最大轮数、失败处理、最终回答生成与结构化日志

---

## 1. 本节学习目标

完成本节后，你应该能够：

1. 解释普通 LLM chat 和 tool agent 的区别。
2. 理解最小 agent loop 的完整流程。
3. 让模型在“回答用户”和“调用工具”之间做选择。
4. 校验模型选择的工具名称和工具参数。
5. 执行工具，并把工具结果追加回 messages。
6. 设计最大循环轮数，避免 agent 无限循环。
7. 处理模型输出解析失败、工具不存在、参数错误和工具异常。
8. 生成最终回答，而不是只返回工具原始结果。
9. 使用结构化日志记录每一步模型决策和工具调用。

这一节的核心是：**把前面三个子模块连接起来，形成一个可控的最小 agent。**

```text
子模块 1：会调用一次 LLM
子模块 2：能解析模型结构化输出
子模块 3：有可调用、可校验、可测试的工具
子模块 4：把它们连成 agent loop
```

---

## 2. 什么是 Agent Loop

### 定义

Agent loop 是一个循环控制流程。它让模型不只是回答一次，而是可以多次观察上下文、做决策、调用工具、读取工具结果，直到生成最终回答。

最小流程：

```text
user input
-> build messages
-> model decides whether to call a tool
-> validate tool arguments
-> execute tool
-> append tool result to messages
-> model decides next step
-> repeat until final answer or max rounds
```

### 和普通 Chat 的区别

普通 chat：

```text
用户输入
-> 模型回答
-> 结束
```

agent loop：

```text
用户输入
-> 模型决策
-> 可能调用工具
-> 观察工具结果
-> 再次决策
-> 最终回答
```

一句话记忆：

```text
普通 chat 是一次生成；agent loop 是带控制边界的多步执行。
```

---

## 3. 为什么需要 Agent Loop

单次 LLM 调用只能完成模型自己能完成的任务。  
但真实 agent 往往需要外部能力：

- 计算准确结果。
- 搜索本地文件。
- 查询网页摘要 mock。
- 新增或完成待办事项。
- 根据工具结果继续推理。

例如用户说：

```text
搜索 notes 目录里有没有提到 RAG evaluation，然后把找到的结果总结一下。
```

这不是一次普通 chat 能稳定完成的任务。更合理的过程是：

```text
模型判断需要 file_search
-> 工具搜索文件
-> 模型读取搜索结果
-> 模型总结给用户
```

agent loop 的价值是把模型推理和工具执行组织成一个可控过程。

---

## 4. Agent Loop 的核心角色

### 用户

用户给出自然语言任务，例如：

```text
帮我计算 19 * 23 + 7。
```

### 模型

模型负责理解任务和做决策：

- 是否需要工具。
- 需要哪个工具。
- 工具参数是什么。
- 工具结果是否足够。
- 最终如何回答用户。

### 工具

工具负责执行确定性动作：

- 计算器负责算表达式。
- 文件检索负责搜索本地文件。
- 网页摘要 mock 负责返回预设摘要。
- 待办工具负责新增、查看、完成任务。

### Agent Loop 控制器

这是你写的 Python 控制代码。它负责：

- 构造 messages。
- 调用模型。
- 解析模型输出。
- 校验工具名称。
- 校验工具参数。
- 执行工具。
- 追加工具结果。
- 控制最大轮数。
- 记录日志。
- 返回最终回答。

模型不是 agent 的全部。  
真正的 agent 是“模型 + 工具 + 控制循环 + 安全边界”。

## 5. 模型决策输出

为了让 agent loop 可控，模型每一步最好输出结构化决策，而不是自然语言。

一个简单决策结构可以是：

```json
{
  "type": "tool_call",
  "tool_name": "calculator",
  "arguments": {
    "expression": "19 * 23 + 7"
  },
  "reason": "用户要求进行精确计算。"
}
```

或者最终回答：

```json
{
  "type": "final_answer",
  "answer": "19 * 23 + 7 的计算结果是 444。"
}
```

这样 agent loop 可以写成：

```text
如果 type == tool_call：执行工具
如果 type == final_answer：结束循环并返回 answer
```

### 为什么不用自然语言判断

如果模型输出：

```text
我觉得应该调用计算器，表达式是 19 * 23 + 7。
```

程序很难稳定解析。  
子模块 2 的 structured output 在这里正式派上用场。

## 6. Messages 如何变化

agent loop 的 messages 会随着循环不断增长。

初始：

```text
system: 你是一个可以调用工具的 agent...
user: 帮我计算 19 * 23 + 7。
```

模型决策：

```text
assistant: {"type": "tool_call", "tool_name": "calculator", ...}
```

工具结果：

```text
tool: {"ok": true, "content": "计算结果是 444", "data": {"value": 444}}
```

再次调用模型：

```text
system: ...
user: ...
assistant: tool_call 决策
tool: 工具结果
```

模型最终回答：

```text
assistant: {"type": "final_answer", "answer": "计算结果是 444。"}
```

### 关键点

工具结果不是直接丢给用户就结束。  
通常要把工具结果放回 messages，让模型根据结果组织最终回答。

---

## 7. Tool Call 决策

### 定义

Tool call 决策是模型选择调用某个工具，并给出参数。

例如：

```json
{
  "type": "tool_call",
  "tool_name": "file_search",
  "arguments": {
    "directory": "notes",
    "query": "RAG evaluation",
    "max_results": 5
  },
  "reason": "用户要求搜索本地笔记。"
}
```

### 工程要求

程序必须检查：

- `type` 是否是允许值。
- `tool_name` 是否在工具白名单里。
- `arguments` 是否是对象。
- 参数是否通过该工具的 Pydantic model 校验。

模型只是提出请求，不能直接执行。

---

## 8. Final Answer 决策

### 定义

Final answer 决策表示模型认为已经有足够信息，可以向用户给出最终回答。

例如：

```json
{
  "type": "final_answer",
  "answer": "我已经添加待办：周五整理 mini-tool-agent README。"
}
```

### 什么时候应该 final answer

常见情况：

- 用户只是普通问答，不需要工具。
- 工具已经返回足够结果。
- 工具失败，但可以向用户说明失败原因。
- 已达到最大轮数，需要给出当前最好回答或错误提示。

### 注意点

不要让 agent 永远等待“更完美的信息”。  
agent loop 必须有结束条件。

---

## 9. 最大循环轮数 max_steps

### 定义

`max_steps` 是 agent loop 允许模型决策和工具调用的最大轮数。

示例：

```python
max_steps = 5
```

### 为什么需要最大轮数

如果没有最大轮数，agent 可能陷入无限循环：

```text
模型调用工具
-> 工具返回结果
-> 模型继续调用同一个工具
-> 工具再次返回结果
-> 模型继续调用...
```

无限循环会导致：

- token 成本失控。
- API 调用变多。
- 用户等待时间过长。
- 程序资源被占用。
- 难以调试。

### 到达最大轮数怎么办

可以选择：

- 返回错误：`达到最大步骤数，未能完成任务。`
- 要求模型基于已有信息生成最终回答。
- 返回最后一次工具结果和失败原因。

本阶段建议简单处理：

```text
超过 max_steps 后，返回一个清晰错误结果，并记录日志。
```

---

## 10. 工具白名单

### 定义

工具白名单是允许模型调用的工具集合。  
模型只能调用注册表中存在的工具。

例如：

```python
allowed_tools = {
    "calculator",
    "file_search",
    "web_summary_mock",
    "todo",
}
```

### 为什么需要白名单

模型可能输出不存在或危险的工具名：

```json
{
  "type": "tool_call",
  "tool_name": "run_python_code",
  "arguments": {
    "code": "..."
  }
}
```

程序必须拒绝。

白名单的价值：

- 防止任意函数调用。
- 限制 agent 能力边界。
- 让系统行为可预测。
- 方便测试和审计。

---

## 11. 工具参数校验

### 定义

工具参数校验是指在执行工具前，用对应工具的 Pydantic model 检查参数是否合法。

例如：

```python
CalculatorArgs.model_validate(arguments)
```

### 为什么不能跳过

模型可能输出：

```json
{
  "tool_name": "calculator",
  "arguments": {
    "expr": "19 * 23 + 7"
  }
}
```

但计算器工具需要的是 `expression`，不是 `expr`。

如果不校验，工具可能报错或行为异常。  
校验失败应该变成结构化错误，而不是直接崩溃。

---

## 12. 工具异常捕获

### 常见工具异常

- 计算器表达式非法。
- 除以 0。
- 文件路径越权。
- 文件读取失败。
- 待办任务不存在。
- mock URL 未命中。

### 处理方式

工具异常应该被捕获并转换成 `ToolResult(ok=False)`。

例如：

```python
ToolResult(
    ok=False,
    content="工具执行失败。",
    error="表达式包含不允许的语法：Call"
)
```

### 为什么重要

工具失败不是系统崩溃。  
它只是 agent loop 中的一次观察结果。

模型可以基于失败结果：

- 向用户解释。
- 换一个工具。
- 请求用户补充信息。
- 给出最终失败说明。

---

## 13. 模型输出解析失败

### 定义

模型输出解析失败是指模型没有输出符合预期 schema 的 JSON 决策。

例如：

```text
我应该调用计算器。
```

而程序期望：

```json
{
  "type": "tool_call",
  "tool_name": "calculator",
  "arguments": {}
}
```

### 处理方式

可以设计：

```text
第一次解析失败
-> 把错误反馈给模型
-> 要求重新输出合法 JSON
-> 如果仍失败，结束并返回错误
```

本阶段可以保持简单：

- 解析失败时记录日志。
- 返回清晰错误。
- 或者做一次重试。

### 和子模块 2 的关系

子模块 2 学的是“如何解析和校验模型输出”。  
子模块 4 要把这个能力放进循环里。

---

## 14. 最终回答生成

### 为什么不能只返回工具结果

工具结果通常偏机器化：

```json
{
  "ok": true,
  "data": {
    "value": 444
  }
}
```

用户更希望看到：

```text
19 * 23 + 7 的计算结果是 444。
```

所以工具执行完后，通常要让模型基于工具结果生成自然语言最终回答。

### 最终回答的来源

最终回答可以来自：

- 模型直接回答。
- 模型整合一个工具结果。
- 模型整合多个工具结果。
- 工具失败后的解释。

### 结束条件

agent loop 通常在以下情况结束：

- 模型输出 `final_answer`。
- 达到 `max_steps`。
- 出现不可恢复错误。
- 用户请求无法处理。

## 15. 结构化日志

### 定义

结构化日志是指用固定字段记录 agent 每一步行为，而不是随意打印字符串。

示例：

```json
{
  "trace_id": "abc123",
  "step": 2,
  "event": "tool_call",
  "tool_name": "calculator",
  "arguments": {
    "expression": "19 * 23 + 7"
  },
  "ok": true,
  "latency_ms": 3.4
}
```

### 应该记录什么

建议记录：

- `trace_id`
- `step`
- `event`
- 模型决策类型
- 工具名称
- 工具参数摘要
- 工具是否成功
- 错误信息
- 耗时
- 最终状态

### 为什么需要日志

agent loop 的问题通常不是一眼能看出来的。

日志可以帮助你回答：

- 模型为什么选了这个工具？
- 工具参数是什么？
- 工具有没有失败？
- 失败发生在哪一步？
- 是否超过最大轮数？
- 用户最终看到的回答来自哪里？

## 16. Observe、Decide、Act

agent loop 常被理解为：

```text
Observe -> Decide -> Act -> Observe -> Answer
```

### Observe：观察

模型读取：

- 用户输入。
- 历史 messages。
- 工具返回结果。
- 错误信息。

### Decide：决策

模型决定：

- 是否调用工具。
- 调用哪个工具。
- 参数是什么。
- 是否最终回答。

### Act：行动

程序执行：

- 工具名称检查。
- 参数校验。
- 工具调用。
- 结果记录。

### Answer：回答

模型基于已有信息输出最终回答。

注意：  
`Act` 应该由程序完成，不应该由模型直接执行任意代码。

## 17. 什么时候让模型决策

适合让模型决策的场景：

- 判断用户意图。
- 判断是否需要工具。
- 从多个工具中选择一个。
- 从自然语言中抽取参数。
- 根据工具结果组织最终回答。

例如：

```text
用户说“帮我查一下 notes 里有没有 RAG evaluation”
模型判断应该调用 file_search
```

## 18. 什么时候写确定性代码

适合写确定性代码的场景：

- 参数校验。
- 工具白名单检查。
- 文件路径安全检查。
- 计算数学表达式。
- 写入待办事项。
- 最大轮数控制。
- 错误捕获。
- 日志记录。

例如：

```text
模型可以决定要搜索文件，但不能决定是否允许读取项目外路径。
```

一句话记忆：

```text
让模型做语义判断，让代码做安全边界和确定性执行。
```

## 19. 两次工具调用任务

验收标准要求能演示一次包含 2 次工具调用的任务。

示例：

```text
添加一个待办：整理 README。然后查看当前待办列表。
```

可能流程：

```text
step 1: 模型调用 todo add
step 2: 工具返回新增成功
step 3: 模型调用 todo list
step 4: 工具返回待办列表
step 5: 模型生成最终回答
```

这能验证 agent loop 是否真的支持多步执行，而不是只支持一次工具调用。

## 20. 不建议本阶段做什么

学习路线明确不建议本阶段做：

- 多 agent 协作。
- 长期记忆。
- 复杂规划器。
- 接真实浏览器自动化。
- 过早引入 LangGraph。

原因是：  
你现在的目标是理解最小 agent loop 的控制边界。  
如果太早引入复杂框架或多 agent，很容易把基础问题藏起来。

本阶段应该先把这件事做扎实：

```text
模型输出决策
-> 解析决策
-> 调用工具
-> 追加结果
-> 循环控制
-> 最终回答
```

## 21. 本节常见误区

### 误区 1：agent loop 就是 while True

不是。  
agent loop 必须有最大轮数、结束条件、错误处理和日志。

### 误区 2：模型说调用什么就调用什么

不行。  
工具名必须在白名单里，参数必须校验。

### 误区 3：工具结果直接就是最终回答

有时可以，但通常需要模型整合成用户友好的回答。

### 误区 4：工具失败就是系统失败

工具失败是正常观察结果。  
agent 可以基于失败结果继续决策或解释给用户。

### 误区 5：越复杂越像 agent

不是。  
一个简单、可测、可控的 agent loop，比复杂但不可调试的系统更有价值。

## 22. 复习小抄

| 概念 | 一句话理解 | 工程作用 |
| --- | --- | --- |
| Agent Loop | 模型多步决策和工具调用循环 | 让 agent 能观察、行动、再观察 |
| Tool Call | 模型请求调用工具 | 把语义决策交给模型 |
| Final Answer | 模型输出最终回答 | 结束循环并返回用户 |
| max_steps | 最大循环轮数 | 防止无限循环和成本失控 |
| 工具白名单 | 只允许调用注册工具 | 控制安全边界 |
| 参数校验 | 执行工具前校验 arguments | 防止错误输入进入工具 |
| 工具异常捕获 | 把工具错误转成结构化结果 | 防止 agent 崩溃 |
| 输出解析失败 | 模型决策不是合法结构 | 触发错误处理或重试 |
| 结构化日志 | 固定字段记录执行过程 | 支持调试和复盘 |
| Observe | 观察用户输入和工具结果 | 给模型决策上下文 |
| Decide | 模型选择下一步 | 判断工具调用或最终回答 |
| Act | 程序执行工具 | 完成确定性动作 |
| 确定性代码 | 可预测的程序逻辑 | 做校验、安全、执行和控制 |

## 24. 下一步实践

下一步可以开始实现子模块 4 的命令行版 `mini-tool-agent`：

```text
1. 定义模型决策结构：
   - ToolCallAction
   - FinalAnswerAction

2. 编写 agent prompt：
   - 告诉模型可用工具
   - 要求模型输出结构化 JSON
   - 明确何时 tool_call，何时 final_answer

3. 实现 agent loop：
   - 初始化 messages
   - 循环调用模型
   - 解析模型输出
   - 执行工具
   - 追加工具结果
   - 到 final_answer 或 max_steps 停止

4. 接入子模块 3 的工具注册表：
   - calculator
   - file_search
   - web_summary_mock
   - todo

5. 增加失败处理：
   - 工具不存在
   - 参数错误
   - 工具异常
   - 模型输出解析失败
```
