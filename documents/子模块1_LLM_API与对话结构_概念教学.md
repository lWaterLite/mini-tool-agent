## 2. LLM API

### 定义

LLM API 是应用程序调用大语言模型的接口。你的程序把用户输入、系统指令、历史对话和参数配置发送给模型，模型返回回答、结构化结果，或者一段段流式 token。

可以把它理解为一个远程推理服务：

```text
你的程序
-> 构造请求 messages + 参数
-> 调用模型 API
-> 接收模型响应
-> 解析、展示、记录或进入下一步逻辑
```

### 作用

LLM API 让你的应用具备自然语言能力，例如：

- 问答
- 总结
- 分类
- 意图识别
- 代码生成
- 文档改写
- 工具调用决策
- agent loop 中的观察、决策和回答

### 工程注意点

LLM API 不是聊天窗口，而是工程系统里的一个外部依赖。因此需要考虑：

- API key 如何配置。
- 模型名称如何切换。
- 请求失败如何处理。
- 返回内容如何解析。
- 请求和响应如何记录日志。
- token 成本如何控制。
- 输出格式如何约束。

## 3. 一次 LLM 请求的基本结构

### 定义

一次普通 LLM 请求通常包含模型名称、消息列表和生成参数。

示例：

```python
response = client.chat.completions.create(
    model="your-model-name",
    messages=[
        {"role": "system", "content": "你是一个技术助教。"},
        {"role": "user", "content": "什么是 agent loop？"},
    ],
    temperature=0.3,
    max_tokens=800,
)
```

### 常见字段

`model`：选择使用哪个模型。不同模型的能力、速度、价格、上下文长度可能不同。

`messages`：对话消息列表。模型会根据这些消息理解任务、身份、约束和上下文。

`temperature`：控制输出随机性。数值越低越稳定，数值越高越发散。

`max_tokens`：限制模型最多输出多少 token，用于控制长度、耗时和成本。

`stream` 或 `streaming`：是否开启流式输出。开启后模型会边生成边返回。

### 作用

这些字段共同决定模型如何理解任务、怎样生成内容、输出多长、输出是否稳定，以及你的程序如何接收结果。

## 4. messages

### 定义

`messages` 是一组有顺序的对话消息。模型并不真正记住你们之前聊过什么，而是在每次请求时读取你发送给它的上下文。

示例：

```python
messages = [
    {"role": "system", "content": "你是一个严谨的 Python 技术助教。"},
    {"role": "user", "content": "解释一下 FastAPI 的 Depends。"},
    {"role": "assistant", "content": "Depends 是 FastAPI 的依赖注入机制..."},
    {"role": "user", "content": "能给一个简单例子吗？"},
]
```

### 作用

`messages` 决定模型能够看到哪些信息，包括：

- 当前用户问题
- 系统规则
- 开发者约束
- 历史对话
- 工具调用结果
- 需要参考的上下文

### 工程注意点

- messages 越长，token 成本越高。
- 历史消息太多会挤占上下文窗口。
- 消息顺序会影响模型理解。
- 重要规则应该放在高优先级消息里，而不是混在用户输入中。
- agent loop 中会不断追加模型决策和工具结果，因此更要管理上下文长度。

## 5. 消息角色：system、developer、user、assistant、tool

### system

`system` 是系统级指令，通常用于定义模型的总体行为、身份、边界和输出风格。

示例：

```text
你是一个严谨、耐心的技术助教。回答时先给定义，再说明作用，最后给例子。
```

作用：

- 设定模型身份。
- 设定回答风格。
- 设定基本规则。
- 提供高优先级行为约束。

### developer

`developer` 是开发者级指令，用于表达应用开发者希望模型遵守的规则。不是所有模型平台都会显式支持这个角色，但在概念上它代表应用层约束。

示例：

```text
回答必须简洁。如果信息不确定，请明确说明不确定。
```

作用：

- 约束应用内的模型行为。
- 统一产品体验。
- 和用户输入区分开，避免用户轻易覆盖应用规则。

### user

`user` 是用户输入，也就是最终用户提出的问题或任务。

示例：

```text
请解释一下 temperature 参数。
```

作用：

- 表达当前任务。
- 提供用户问题。
- 提供需要处理的文本、命令或上下文。

### assistant

`assistant` 是模型之前的回复。多轮对话时，历史 assistant 消息会被放回 messages，帮助模型延续上下文。

作用：

- 保留模型之前的回答。
- 让模型知道前面对话已经说过什么。
- 在 agent 中也可能表示模型之前做出的决策。

### tool

`tool` 是外部工具执行后的返回结果。在 agent 场景中，模型可能决定调用计算器、文件搜索、网页摘要等工具，工具执行完后结果会作为 `tool` 消息放回 messages。

作用：

- 把外部世界的信息反馈给模型。
- 让模型基于工具结果继续推理。
- 支持“观察 -> 决策 -> 行动 -> 观察 -> 回答”的 agent loop。

### 简单记忆

```text
system / developer: 规则和身份
user: 用户任务
assistant: 模型回复或决策
tool: 外部工具执行结果
```

## 6. temperature

### 定义

`temperature` 控制模型输出的随机性。数值越低，输出越稳定；数值越高，输出越发散。

### 作用

低 temperature，例如 `0` 到 `0.3`：

- 更稳定。
- 更适合事实问答、代码解释、分类、结构化输出。
- 同一个问题多次调用，答案差异较小。

中等 temperature，例如 `0.5` 到 `0.8`：

- 有一定变化。
- 适合教学解释、表达润色、头脑风暴。

高 temperature，例如 `1.0` 以上：

- 更有创造性。
- 更容易跑偏。
- 更容易出现格式不稳定或编造内容。

### 工程注意点

对于 `mini-tool-agent` 这种工程项目，早期建议使用较低 temperature，例如 `0` 到 `0.3`。因为你需要稳定、可测试、可调试的行为。

## 7. max tokens

### 定义

token 是模型处理文本的基本单位。`max_tokens` 用来限制模型最多输出多少 token。

### 作用

- 控制回答长度。
- 控制接口费用。
- 控制响应耗时。
- 避免模型输出过长。
- 让产品体验更可控。

### 工程注意点

不同任务适合不同的限制：

- 简短分类：可以设置较小的 `max_tokens`。
- 技术解释：可以设置中等长度。
- 长文总结或报告生成：需要更大的限制。

如果 `max_tokens` 太小，模型可能回答不完整；如果太大，可能增加成本和延迟。

---

## 8. streaming

### 定义

`streaming` 指模型不是等完整答案生成完再返回，而是边生成边把内容返回给客户端。

普通输出：

```text
等待几秒
一次性看到完整回答
```

流式输出：

```text
模型生成一点
客户端显示一点
用户持续看到反馈
```

### 作用

- 提升交互体验。
- 适合聊天界面。
- 适合命令行助手。
- 让用户更早看到响应。

### 工程注意点

streaming 会增加实现复杂度：

- 需要处理一段段增量内容。
- 如果要保存完整回复，需要自己累积 chunks。
- 错误处理更复杂。
- Web API 层可能需要 SSE、WebSocket 或流式响应。

子模块 1 阶段只需要先体验普通输出和流式输出的差异。

## 9. 上下文长度

### 定义

上下文长度是模型一次请求中最多能处理的 token 数，包括输入和输出。

一次请求中的上下文通常包含：

```text
system / developer 指令
+ 历史 user 消息
+ 历史 assistant 消息
+ tool 结果
+ 当前 user 输入
+ 模型即将输出的内容
```

### 作用

上下文长度决定模型一次能够“看见”多少信息。

### 工程注意点

如果上下文超限，可能会出现：

- 请求失败。
- 旧消息被截断。
- 模型忘记前文。
- 输出质量下降。
- 成本明显升高。

Agent 项目尤其需要注意上下文长度，因为 agent loop 会不断追加模型决策、工具调用结果和中间观察。

## 10. token 成本

### 定义

LLM API 通常按 token 计费。费用一般分为输入 token 和输出 token。

```text
总费用 = 输入 token 费用 + 输出 token 费用
```

### 作用

理解 token 成本可以帮助你设计更经济的系统。

### 工程注意点

- 历史对话越长，输入成本越高。
- 模型回答越长，输出成本越高。
- agent 多轮调用模型，成本会叠加。
- 工具返回内容太长，会增加后续请求成本。
- prompt 中无关内容越多，浪费越明显。

所以工程上需要控制上下文、压缩工具返回、限制输出长度，并记录关键调用成本。

## 11. prompt

### 定义

prompt 是你给模型的指令、任务、上下文和输出要求的总和。它不只是用户那一句话，也包括 system prompt、developer prompt、用户输入、示例和格式约束。

示例：

```text
你是一个技术助教。
请用适合初学者的方式解释概念。

要求：
1. 先给定义
2. 再说明作用
3. 最后给一个简单例子

用户问题：什么是 streaming？
```

### 作用

prompt 用来引导模型：

- 以什么身份回答。
- 回答什么任务。
- 使用什么格式。
- 遵守什么边界。
- 面对不确定信息时如何处理。

### 工程注意点

prompt 是软约束，不是强制逻辑。模型可能没有完全遵守 prompt，因此后续还需要结构化输出、Pydantic 校验、重试机制和测试来兜底。

## 12. 稳定、可控、可调试的 prompt

### 稳定

稳定指同类输入能够得到相对一致的输出。

例如每次解释概念时都按：

```text
定义
作用
例子
注意点
```

组织内容。

### 可控

可控指你可以通过明确规则影响模型行为。

例如：

- 限制回答长度。
- 指定输出格式。
- 要求不确定时说明不确定。
- 要求不要编造不存在的 API。

### 可调试

可调试指出问题后能定位原因。

常见问题来源包括：

- system prompt 不够明确。
- 用户输入太模糊。
- temperature 太高。
- 上下文过长。
- 输出格式要求不清晰。
- 示例和规则互相矛盾。

### 一个实用 prompt 结构

```text
角色：你是谁
任务：你要做什么
输入：用户提供了什么
约束：不能做什么，必须做什么
格式：按什么结构输出
边界：不确定时如何处理
```

## 13. 为什么不能只依赖自然语言 prompt 控制输出

### 原因

LLM 是概率生成模型，不是传统确定性函数。即使你写了“请一定输出 JSON”，模型也可能输出：

```text
好的，下面是 JSON：
{
  "intent": "search"
}
```

人类能看懂，但程序直接解析时可能失败。

### 工程结论

prompt 负责引导，代码负责兜底。

在工程项目中，需要组合使用：

- prompt 引导模型。
- JSON schema 限制结构。
- Pydantic 校验输出。
- 重试机制处理失败。
- 日志记录请求和响应。
- 测试覆盖关键边界。

这也是后续子模块 2 要学习 structured output 和 Pydantic 的原因。

## 14. 模型客户端初始化

### 定义

模型客户端是代码中用于调用 LLM API 的对象。它通常负责读取配置、发送请求和接收响应。

示例：

```python
import os
from openai import OpenAI

client = OpenAI(api_key=os.getenv("OPENAI_API_KEY"))
```

### 作用

- 管理 API key。
- 配置模型服务地址。
- 调用模型接口。
- 统一请求行为。
- 方便后续测试和替换模型供应商。

### 工程注意点

不要在项目中到处重复初始化客户端。更好的方式是封装到配置层或客户端模块中，例如：

```text
app/
  core/
    config.py
  agent/
    client.py
```

## 15. 环境变量管理

### 定义

环境变量是在运行程序时注入的配置，例如 API key、模型名称、日志级别等。

示例：

```text
OPENAI_API_KEY=your_api_key_here
MODEL_NAME=your_model_name_here
LOG_LEVEL=INFO
```

### 作用

- 避免把 API key 写死在代码里。
- 让本地、测试、生产环境使用不同配置。
- 方便 Docker、CI 和服务器部署。
- 降低敏感信息泄露风险。

### 工程注意点

项目中通常应该有：

```text
.env              本地真实配置，不提交到 Git
.env.example      示例配置，可以提交到 Git
```

`.env.example` 只放占位符，不放真实密钥。

## 16. 普通聊天请求

### 定义

普通聊天请求是不涉及工具调用、不强制结构化输出的基础 LLM 调用。

流程：

```text
用户输入
-> 构造 messages
-> 调用 LLM
-> 读取 assistant 回复
-> 输出给用户
```

### 作用

它是后续所有 agent 能力的基础。你需要先打通：

- API key 配置。
- 客户端初始化。
- messages 构造。
- 模型调用。
- 响应读取。
- 错误处理。
- 命令行输入输出。

### 对应练习

子模块 1 的第一个练习是写一个 `ask_llm.py`，从命令行读取用户输入并返回模型回答。

## 17. prompt 模板

### 定义

prompt 模板是把固定指令和变量输入分离，方便复用和维护。

示例：

```python
SYSTEM_PROMPT = "你是一个严谨、耐心的技术助教。"

USER_TEMPLATE = """
请解释下面这个概念：

{concept}

要求：
- 给出定义
- 说明作用
- 给一个例子
"""
```

### 作用

- 复用常见 prompt。
- 减少重复字符串。
- 方便测试。
- 方便修改规则。
- 让 prompt 结构更清晰。
- 以后可以迁移到单独文件或配置中管理。

### 工程注意点

prompt 模板不要过度复杂。早期先保持清晰、可读、容易修改，等项目变大后再考虑更系统的 prompt 管理方式。

## 18. 简单错误处理

### 定义

简单错误处理是指在调用 LLM API 时，对常见失败情况给出清晰反馈，而不是让程序直接崩溃。

### 常见错误

- API key 没有配置。
- 网络请求失败。
- 请求超时。
- 模型名称错误。
- 配额不足。
- 参数错误。
- 模型返回为空。
- 流式输出中断。

### 作用

- 提升调试效率。
- 改善用户体验。
- 避免程序异常退出。
- 为后续日志和测试打基础。

### 初期处理方式

子模块 1 阶段可以先做到：

- 用户输入为空时，提示重新输入。
- API key 缺失时，输出清晰错误。
- 请求失败时，打印错误原因。
- 模型返回为空时，给出兜底提示。

后续进入 agent loop 后，再扩展为统一异常类型、结构化日志和失败重试。

## 19. 子模块 1 的整体认知框架

你可以把 LLM 调用拆成七层：

```text
配置层：API key、模型名、参数
输入层：system / developer / user / messages
调用层：client 发起请求
输出层：assistant 回复
体验层：普通输出或 streaming
控制层：temperature、max tokens、上下文管理
调试层：日志、错误信息、请求记录
```

后续所有 agent 能力，都是在这个基础上叠加：

```text
普通 LLM chat
-> structured output
-> tool calling
-> agent loop
-> FastAPI 服务
-> 测试、日志、Docker、README
```

7. 为什么不能只靠 prompt 要求模型输出固定格式？
8. 为什么 API key 不能写死在代码里？
9. prompt 模板相比直接拼字符串有什么优势？
10. 如果 API 请求失败，最小版本的程序应该如何反馈？
